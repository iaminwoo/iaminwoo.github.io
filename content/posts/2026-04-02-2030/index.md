## 1) HashMap 기초 : 해싱 Hashing

### 1. HashCode 를 인덱스로 변환하는 원리

HashMap 역시 배열을 활용하는 자료구조이다.
**Key 와 Value 를 저장**하는데, **Key 를 인덱스로** 바꾸고 **해당 위치에 Value 를 저장**하는 구조이다.

> Hashing : **Key -> ( ) -> Index**

Key 객체를 인덱스로 바꾸기 위해 **객체를 숫자로 바꾸는 작업**이 필요하다.

자바의 모든 객체는 **HashCode** 를 가지는데 이를 이용하면
**모든 객체에 대해 특정 숫자값**을 얻을 수 있다.

```java
String hw = "Hello World!";
System.out.println(hw.hashCode());
/* 결과: -969099747 */
```

이 해시코드는 **객체의 인식번호**로 당연히 **중복될 수도 있다.**
반환값은 Int 로 나오는데 이것을 배열의 인덱스로 바꾸게 된다.

> Hashing : **Key -> ( HashCode ) -> Index**

가장 직관적인 아이디어는 반환된 **HashCode 값을 배열의 길이로 나눈 나머지**를 사용하는 것이다.
그러면 자연스럽게 **배열의 길이에 맞는 인덱스들**이 나오게 된다.

> Hashing : **Key -> ( HashCode -> HashCode % arr.length ) -> Index**

### 2. % 대신 & 를 사용하는 이유

하지만,
이것에는 두가지 문제가 있는데
첫번째는 **HashCode 의 값은 음수**가 나올 수 있다는 점이고,
hashing 작업은 자주 수행되는데 **나머지 계산은 무거운 작업**이라는 것이다.

> AND 비트연산 사용!

이 두가지를 **동시에 해결하는 방법**은 **비트 연산**을 사용하는 것이다.

계산을 쉽게 하기 위해서 **일반적으로 배열의 길이는 2의 제곱수**로 정하며,
**( 배열의 길이 -1 ) 과 HashCode 를 AND 연산**할 것이다.
(2의 제곱수 -1 을 이진수로 바꾸면 언제나 1의 모음이 된다.)

```
// AND 계산 수행
11000110 00111100 10110110 00011101 (-969099747)
00000000 00000000 00000000 00001111 (배열의 길이 16 - 1)
----------------------------------------
00000000 00000000 00000000 00001101 (13)
```

첫번째 문제였던 **HashCode 음수 문제가 해결**되었음을 확인할 수 있다.

그래서 위의 경우 13이라는 **결과값이 index 의 값**이 되고,
HashCode 의 전체 32개 비트 중에 **하위 비트 4개만 사용**되고 있는 것을 볼 수 있다.

> **Hashing 으로 나오는 index 의 값**은 HashCode 이진수의 **하위 비트 자리들!**

두번째로 실제로 **나머지 연산과 비트 연산의 속도 차이**를 비교해보았다.

```java
// 1. 테스트용 해시코드 10만 개를 미리 준비 (객체 생성 비용 제거)
int[] hashCodes = new int[100000];
for (int i = 0; i < hashCodes.length; i++) {
  hashCodes[i] = new Object().hashCode();
}

int arrSize = 16;
int mask = arrSize - 1;
long sum = 0; // 결과값을 저장해서 결과없는 연산을 무시하는 JVM 최적화 방지

// --- 실험 1: 비트 연산 (&) ---
long start = System.nanoTime();
for (int h : hashCodes) {
  sum += (h & mask); // 순수 비트 연산만 수행
}
long end = System.nanoTime();
System.out.println("비트 연산 (&) 순수 시간: " + (end - start) / 1_000_000.0 + " ms");
// ---------

sum = 0; // 초기화

// --- 실험 2: 나머지 연산 (%) ---
start = System.nanoTime();
for (int h : hashCodes) {
  sum += (Math.abs(h) % arrSize); // 순수 나머지 연산만 수행
}
end = System.nanoTime();
System.out.println("나머지 연산 (%) 순수 시간: " + (end - start) / 1_000_000.0 + " ms");
// ---------

// --- 출력 결과 ---
"비트 연산 (&) 순수 시간" : 0.215875 ms  // 10배 빠름!
"나머지 연산 (%) 순수 시간" : 2.001166 ms
```

**비트 연산이 나머지 연산보다 10배 빠른 것**을 확인할 수 있다.

### 3. 보조 해시 함수 : 모든 비트를 사용하는 XOR 연산

그런데,
여기서 새로운 문제가 발생한다.
**만약 HashCode 의 값이 이상적이지 않고 하위 비트가 겹치는 값들이라면?**
(메모리 위치 등의 특정 값들이거나 의도적으로 충돌을 유발하려는 경우)

예를들어 13, 29, 45, 61, ... 의 값들은 AND 연산 결과가 언제나 13이 나오게 되고
해당 키를 가진 값들은 **같은 인덱스에서 충돌**이 일어나게 된다.

그래서 **상위 비트들**이 하위비트들에 **영향을 미칠 수 있도록** (모든 비트들이 결과에 영향을 주도록),
32비트 HashCode 값을 반으로 접어 **상위, 하위 16비트를 XOR 연산**을 하는데
이를 **보조 해시 함수**라고 한다.

참고로 AND, OR 연산은 0, 1 둘중에 하나가 결과에 큰 영향을 미치는 반면
**XOR 연산**은 **양쪽의 값들이 모두 동일한 영향**을 준다.

```java
HashCode ^ (HashCode >>> 16)
```

위의 값을 예시로 들면

```
// XOR 계산 수행
11000110 00111100 10110110 00011101 (-969099747)
00000000 00000000 11000110 00111100 (상위 비트 이동)
----------------------------------------
11000110 00111100 01110000 00100001
```

이 경우 **새로운 하위 비트** 4자리를 사용하기 때문에 새로운 인덱스는 1이 된다.

HashCode 가 다양하게 나오는 이상적인 경우에는 결과에 큰 영향은 없지만
**XOR 연산 자체의 비용은 거의 없기 때문에 안전을 위해 사용한다.**

> 안전을 위한 보험 **보조 해시 함수**!

## 2) HashMap 구현 : 충돌 관리

### 1. 해시 충돌의 해결 LinkedList

> Hashing 의 결과가 같아지는 것 = 충돌

아무리 해싱을 다양하고 복잡하게 구현을 한다고 해도
해싱의 결과인 **인덱스는 반드시 중복**되게 되어 있다.

이렇게 **인덱스가 중복**되어 **다수의 요소가 하나의 인덱스**를 향하는 현상을 **충돌**이라고 하고,
Java HashMap 에서는 해당 인덱스의 노드를 LinkedList 로 만들어 처리한다.

단순하게 예시를 들면,
**주차장**에 차를 주차하는데 **차량마다 특정 공간**을 부여받는다고 생각해보자.

이 주차장에 들어갈 때는 **차량번호를 주고 주차 공간을 부여**해주면
반드시 그 장소에 주차해야 한다.

그리고 떠날 때는 **내 차가 'A-1' 에 주차**가 되어 있다면
**'A-1' 칸에 가서 찾으면 된다.** (**효율적인 O(1) 방식**)

그런데 만약 'A-1' 칸에 갔는데 **다른 차가 이미 있다면** (**충돌**)
그 차 뒤로 **이중 주차**를 해야 한다. (**LinkedList 방식**)

그리고 차를 찾을 때는 그 칸에 있는 차들이 **순서대로 나오면서** 내 차를 찾아야 한다.
(**O(n)**, **해시 충돌**로 인한 성능 저하)

### 2. HashMap 기초 구현

그러면 본격적으로 **HashMap** 을 구현해보자.

가장 먼저 **Key 와 Value** 를 담을 **Node 클래스**가 필요하다.

```java
private static class Node<K, V> {
  final K key;        // Key 값
  V value;            // Value 값
  Node<K, V> next;    // LinkedList 로 만들기 위한 next 포인터

  private Node(K k, V v) {
    key = k;
    value = v;
  }
}
```

그리고 **Key 값을 인덱스로** 바꾸는 **Hashing 작업**이 필요하다.

```java
private int getIndex(Object key) {
  int hashcode = key.hashCode();

  hashcode = hashcode ^ (hashcode >>> 16);   // XOR 보조 해싱

  return hashcode & (table.length - 1);      // AND 연산으로 인덱스 추출
}
```

이 둘을 합쳐서 **요소를 추가하는 메서드**를 작성하면,

```java
private Node<K, V>[] table = (Node<K, V>[]) new Node[16];  // 배열의 길이 = 2의 제곱수

public void put(K key, V value) {
  Node<K, V> newNode = new Node<>(key, value);
  int index = getIndex(key);  // 해싱으로 인덱스 추출

  if (table[index] == null) {
    // 1. 만약 해당 위치가 비어있다면 그냥 삽입
    table[index] = newNode;
  } else {
    // 2. 만약 해당 위치에 이미 노드가 있다면 (충돌)
    Node<K, V> cur = table[index];

    // 연결된 노드들을 확인하며
    while (true) {
      if (cur.key.equals(key)) {
        // 2-1. Key 가 같다면 덮어쓰기
        cur.value = value;
        return;
      }

      if (cur.next == null) {
        // 2-2. 노드들 마지막에 삽입
        cur.next = newNode;
        break;
      }
      cur = cur.next;
    }
  }
}
```

## 3) HashMap 의 실제 구현

### 1. Hash 값을 따로 저장하는 Node

위에서 구현했던 Node 와 **실제 HashMap 의 Node**가 다른 점은,

```java
static class Node<K,V> implements Map.Entry<K,V> {
  final int hash;   // hash 값을 저장하는 필드
  final K key;
  V value;
  Node<K,V> next;

  // ... 생략
}
```

**hash 값을 저장하는 필드**가 있다는 것이다.

> Node 내부 필드로 hash 값 저장!

숫자의 경우는 자기 자신이 해시값이 되어 계산이 빠르지만,
문자열이나 일반 객체의 경우에는 **해싱 과정이 오래** 걸리기 때문에
매번 hash 값을 불러오지 않고 **따로 저장하여 사용**한다.

이 필드는 가장 먼저 **get 메서드** 사용시 **1차 필터링**으로 사용된다.

```java
if (e.hash == hash && ((k = e.key) == key || (key != null && key.equals(k))))
```

해당 노드의 Key **객체가 같은지** 비교하는 것보다
hash **숫자값을 비교**하는 것이 **더 빠르기 때문**이다.

> 객체 equals 보다 숫자값 비교가 빠름!

### 2. 임계치 초과시 발생하는 확장

그리고 hash 필드는 **HashMap 의 크기를 키우는 Resize 메서드**에서도 사용되는데,
HashMap 은 **전체 크기의 75%가 차면 확장을 고려**한다.

이 비율이 **너무 낮다면** HashMap 은 **자주 확장**을 하게 되는데,
공간이 많기 때문에 **요소들의 충돌 확률이 낮아지게** 된다.
하지만 그만큼 빈 공간도 많기 때문에 **메모리 낭비**가 심해지게 된다.

반대로 이 비율이 너무 높다면 **메모리는 알차게** 사용할 수 있겠지만,
**충돌이 자주** 일어나 성능이 안좋아지게 된다.

그래서 Java 개발자들은 **시공간적 효율의 적정 비율**을 **75% 로 설정**했다.

```java
static final float DEFAULT_LOAD_FACTOR = 0.75f;
```

그리고 HashMap 에 **새로운 요소가 추가**되었는데
이 비율을 넘겼다면 **확장을 진행**한다.

```java
// size : 현재 HashMap 의 크기, threshold : 확장 기준 크기
if (++size > threshold) resize();
```

위에서 이야기했던 것처럼 확장은 **현재 크기의 2배로** 이루어진다.
이전의 **인덱스를 구하는 것과 비슷한 연산**을 진행하는데,

```java
if ((e.hash & oldCap) == 0)
```

이전에 인덱스를 구하는데 사용했던 비트들 말고 **(hash & (table.length - 1))**
**다음 비트를 사용**하게 된다. **(hash & table.length)**

```
< 기존 인덱스 >
1010 1101 (hash = 173)         1011 1101 (hash = 189)
0000 1111 (table.length - 1)   0000 1111 (table.length - 1)
-------------                  -------------
0000 1101 (index = 13)         0000 1101 (index = 13)

< 새 인덱스 >
1010 1101 (hash = 173)         1011 1101 (hash = 189)
0001 0000 (table.length)       0001 0000 (table.length)
-------------                  -------------
0000 0000 (newIndex = index)   0001 0000 (newIndex = table.length + index)
```

그렇기 때문에 확장 후 요소는 **기존의 인덱스 (앞 부분) 로 가거나,**
**뒤쪽 반의 현재 인덱스 (뒷부분, 기존 크기 + 기존 인덱스) 로** 가게 된다.

그래서 **최악의 경우** 확장을 했는데도 **모든 요소가 한 칸에 그대로** 모여있을 수도 있다.

이런 최악의 경우를 막기 위해 처음 해시를 구할 때
상위와 하위 비트를 섞는 보조 해시 함수 과정을 거치는 것이지만,
기본적으로 HashMap 은 **요소들을 완벽하게 나누는 노력**보다
잘 나눠지고 아니면 다시 확장을 하며 나누는 **속도를 선택**한 것이다.

> 비트 확인으로 속도를 높이는 Resize 메서드!

### 3. 수상하게 많은 충돌에 대비하는 트리화

이런 식으로 요소들이 많아지면 충돌을 막기 위해 확장을 하는데,
아주 흔치 않은 경우이지만 **애초에 해시가 이상하게 들어오면**
**공간이 충분한데도 충돌**이 많이 일어날 수 있다.

공식 주석에 의하면,
해시코드가 충분히 효율적이라면,
이상적으로 75% 확장을 하는 HashMap 에서
**요소들의 분포**는 포아송 분포를 따른다고 한다.

그리고 그 값들은 다음과 같은데

- 0: 0.60653066
- 1: 0.30326533
- 2: 0.07581633
- 3: 0.01263606
- 4: 0.00157952
- 5: 0.00015795
- 6: 0.00001316
- 7: 0.00000094
- 8: 0.00000006

특정 버킷이 빈 공간일 확률은 60.6%,
요소 하나가 들어 있을 확률은 30.3%, ... 등등이다.

HashMap 은 '**확률상 불가능**' 의 기준을 0.000006%인 **8개**로 잡는다.

**하나의 버킷에 요소들이 8개**가 들어 있다면,
일반적인 상황이 아니라 **해시함수가 아주 이상**하거나 **의도적인 공격**으로 간주하는 것이다.

그리고 이런 이상한 **버킷 하나 때문에 전체를 확장을 할 수는 없고**
확장을 하더라도 해당 버킷은 다시 가득 찰 수 있기 때문에
**그 부분을 Tree 구조로 변경**시킨다. (정확히는 **Red-Black Tree**)

다만 하나의 버킷에 노드가 8개가 된다고 바로 트리화하지는 않고,
**전체 테이블의 크기가 64 미만**일 때는 **크기가 부족하다고 판단**하여
확장을 한번 더 시도하게 된다.

주차장 예시로 보면 **주차장 전체**는 비어있고 충분히 넓은데
**특정 칸**에만 차들이 너무 많이 줄이 서있다면,
주차장 확장보다 그 칸에만 **주차타워**를 세우는 것이 적절할 것이다.

이때 사용하는 TreeNode 는 **일반 Node 보다 두배의 크기**를 가지고 있어서,
요소 삭제나 확장을 통해 **Node 의 수가 6으로 내려오면 다시 일반 리스트로 전환**한다.

6에서 8까지의 2라는 여유를 둔 것은 **리스트를 트리로** 만들거나
반대의 작업이 **상당히 무거운 작업**이기 때문이다.

> 특정 버킷에 요소들이 몰린다면 **해당 버킷을 트리화**한다!

## 4) 마치며 : HashMap 의 시간복잡도와 개발자의 책임감

### 1. 각 작업별 시간복잡도

HashMap 의 각 작업의 **시간복잡도는 이상적인 상황에서 모두 O(1)** 이다.
요소들이 테이블에 적당하게 퍼져있어서 특정 Key 로 Value 를 바로 찾을 수 있는 상태이다.

하지만 **해시 충돌**이 일어나기 시작하면 **O(n)** 까지 길어질 수도 있다.
(Red-Black Tree 를 사용하면 **O(log n)** 까지)

### 2. 개발자의 책임감

> Note that using **many keys with the same hashCode**
> is a sure way to **slow down performance** of any hash table.

이처럼 HashMap 의 다양한 최적화에도 불구하고 **모든 작업의 성능**은
**해시코드가 얼마나 무작위로 잘 퍼지는지**에 달려 있다.

그렇기 때문에 HashMap 을 사용할 때는
**내가 사용하는 Key 값이 어떻게 해싱**되는지,
**한 곳에 몰리지는 않는지** 반드시 염두하며 사용해야겠다.
