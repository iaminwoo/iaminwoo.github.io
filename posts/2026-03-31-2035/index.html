1. 개인 블로그를 시작하게 된 계기
#

1-1. 블로그 시작, Velog
#

프로그래머스에서 부트캠프를 했을 때, 개인 블로그를 하는 것이 취업에 유리하다는 말을 많이 들었다.

거기서 몇개의 플랫폼을 추천해주었는데
네이버 블로그나 티스토리, 그리고 개발자들을 위한 블로그라는 Velog 였고
그 중에서 Velog 를 통해서 꽤 많은 게시물을 작성했었다.

Velog 를 사용하면서 아쉬웠던 것은
주된 사용자들이 블로그 작성을 통해 개인의 학습을 증명하려는 취준생들이었고
생성되는 컨텐츠들도 인터넷에서 아주 많이 찾아볼 수 있거나
AI 로 작성된 티가 많이 나는 수준 낮은 게시물들이 많았다.

이것마저도 취업 후나 일정 시간 후에는 더이상 사용되지 않고 버려지는 곳들이 많았다.

그리고 플랫폼이 디자인적으로 그리 마음에 들지 않았다는 것도
Velog 를 떠나게 된 주요 원인 중 하나였다.

1-2. 노션으로 이동
#

그래서 그 다음에는 개인 기록을 목적으로 노션으로 갔다.

그동안의 기록들을 모두 노션으로 옮기는 것도 쉽지는 않았지만
그래도 이 모든 것들을 내가 소유하고 있다는 느낌이 좋았다.

공개되지 않기 때문에 좀 더 글을 편하게 쓸 수 있다는 장점도 있었다.

1-3. 개인 블로그의 필요
#

하지만 결국 포트폴리오 등을 위해서나 다른 사람에게 공유할 만한 가치있는 글을 썼든
노션은 공유하기가 불편하고 뭔가 공식적이지 않은 느낌이 있다.

그래서 개인 블로그를 만들기로 결정을 했는데
이전 Velog 처럼 질보다 양으로 승부하는 쓸모 없는 블로그는 싫다.

뭔가 누가봐도 노력과 연구가 많이 들어간
가치있는 게시물들이 있는 블로그가 되면 좋겠다.

그래서 조금 헤매기는 했지만 Github Pages 를 통해
정적 사이트를 만드는 방식을 선택하였다.

2. 구현 과정
#

2-1. 구현 방식 선택 : Github Pages
#

지금까지 배우고 익힌 기술로도 개인 블로그 정도는 쉽게 만들 수 있었다.
하지만 시간도 많이 들기도 하고 블로그라는게 화려한 구현이 필요한 것도 아니니
좀 더 쉬운 방법인 정적 사이트를 만들기로 했다.

몇가지 선택지가 있긴 했지만
아무래도 이미 경험이 많은 Github 에서 진행하는게 좋을 것 같다는 생각이 들었다.

그런데 Github 에서 딱 만들 수 있는 것이 아니라 호스팅만 해주는 것이었다.
실제적인 사이트의 내용은 정적 사이트 생성기를 통해 만들어야 했다.

2-2. 엔진 선택 : Hugo
#

유명한 생성기는 Jekyll 이나 Hugo 가 있었는데
Hugo 가 더 최신 생성기이기도 하고,
Jekyll 보다 빌드 속도가 언어적 특성으로 인해 빨라서, 페이지 수가 많아졌을때 더 유리해서 선택하였다.

물론 그래서 관련 정보는 상대적으로 부족하여서
문법적인 부분에서 원하는 설정을 찾을 때
Gemini 가 엉뚱한 대답을 많이 해서 어려움을 많이 겪었고
공식 문서를 아주 많이 의지하게 되었다.

2-3. 테마 선택 : Blowfish
#

공식 페이지에 다양한 테마들이 있었는데
그 중에 마음에 드는 Congo 라는 테마를 선택했었다.

그런데 관련 설정에서 최신 Hugo 와 충돌되는 것 같은 부분이 있어
깔끔하게 포기하고 Blowfish 테마로 새로 시작했다.

2-4. 트러블슈팅
#

테마 자체에서 지원하는 기능도 많고 아직은 크게 커스텀 할 부분이 없어서
공식 문서를 열심히 읽고 따라하며 블로그를 만들었다.

여러가지 설정도 바꿔보며 페이지 구성 방식들에 익숙해지고
테스트 글도 작성해보며 로컬에서는 아주 잘 돌아갔었다.

그런데 실제로 배포하기 위해 Github Action 을 돌렸는데,
프로필에 설정한 이미지를 리눅스 서버가 찾지 못하는 문제가 있었다.

한시간 정도 여러가지 설정도 바꿔보고 이미지가 없이 배포도 해보고 했는데,
결국 문제는 이미지 확장자의 대소문자 문제였다.

image = "img/author.jpg"

# 실제 파일명: assets/img/author.JPG

설정 파일에는 소문자 jpg 로 설정을 했는데 실제 이미지가 대문자 JPG 로 되어 있었다.
로컬 환경에서는 대충 비슷하니까 이미지를 가져와서 보여줬는데
리눅스 서버는 대소문자에 훨씬 민감했다.

그래서 결국 확장자는 소문자로 통일하고 성공적으로 배포하였다.

3. 느낀점
#

3-1. 장점
#

지금도 게시물을 vscode 로 작성하고 있는데
아무래도 오래 사용해본 툴이라 더 익숙한 느낌이 있다.

글을 쓰고 있지만 코드를 작성하고 있는 느낌.

그리고 git 으로 작성한 글과 여러 설정들을 한번에 관리할 수 있는 것이 좋다.
실제로 설정들을 변경하면서 문제가 생겼을 때 이전 커밋으로 돌아가 초기화할 수도 있었다.

3-2. 단점
#

조금 익숙해지면 괜찮아질 것 같긴 한데
Velog 나 다른 블로그 플랫폼에 비해서
작성하는 마크다운을 바로 확인하지 못하는 것은 아쉬웠다.

그래서 관련 익스텐션도 설치를 했는데
중간중간 확인하며 마크다운이 잘 적용이 되고 있는지 보며 작성하고 있다.

3-3. 총평
#

직접 페이지를 개발해서 좀 더 애정이 가고
계속 확인하며 불편한 곳은 없는지 확인하고 있다.

전반적으로 기술 블로그의 역할을 하기에는 기능이 아주 충분한 것 같다.
